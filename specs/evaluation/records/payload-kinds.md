# Evaluation payload kinds

This spec lists the payload kinds supported by the first Evaluation
implementation slice.

The key words **MUST**, **MUST NOT**, **SHOULD**, and **MAY** are to be
interpreted as described in BCP 14 when, and only when, they appear in all
capitals.

## Agent-Written Kinds

`qualitymd evaluation data set` **MUST** accept these kinds:

- `EvaluationFrame`
- `AreaEvaluationFrame`
- `RequirementEvaluationFrame`
- `RequirementAssessmentResult`
- `RequirementRatingResult`
- `FactorAnalysisFrame`
- `FactorAnalysisResult`
- `AreaAnalysisFrame`
- `AreaAnalysisResult`

The CLI **MUST** validate each accepted kind before writing it. Validation
**MUST** be derived from one typed source of truth for the kind's structural
fields, required fields, field types, enum values, schema output, and examples.

The CLI **MUST** reject unknown or misspelled fields, wrong field types,
out-of-range enum values, and missing required fields before writing a payload.

The CLI **MUST** resolve every Area ID, Factor ID, Requirement ID, and Rating
Level ID in an accepted payload against the run's model snapshot before writing
it.

## CLI-Owned Kinds

`EvaluationOutputResult` **MUST** be generated by
`qualitymd evaluation report build`.

Agents **MUST NOT** write `EvaluationOutputResult` through
`qualitymd evaluation data set`.

## Required Payload Shape

Frame payloads **MUST** use:

```text
subject
inputs
derivedContext
```

Result payloads **MUST** follow the result shape for their routine.

Each payload kind **MUST** carry enough subject identity for the CLI to derive
its canonical data path without caller-supplied path flags.

Status-bearing outputs **MUST** distinguish completed, incomplete, empty, and
blocked states using the status vocabulary defined by the routine contract.
