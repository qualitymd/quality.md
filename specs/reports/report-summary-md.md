# report-summary.md

`report-summary.md` is the concise human triage artifact generated by
`qualitymd evaluation report build` beside `report.md` and `report.json`.

Shared report-model, trust-boundary, active/superseded, and typed-state
invariants live in [Evaluation records](../evaluation-records.md#report-outputs).
This spec owns the human Markdown shape, labels, and summary-density rules for
the concise artifact.

This document uses BCP 14 keywords only for testable conformance requirements.
The key words "MUST", "MUST NOT", "SHOULD", and "MAY" are to be interpreted as
described in [RFC 2119](../../docs/reference/rfc2119.md) and
[RFC 8174](../../docs/reference/rfc8174.md) when, and only when, they appear in
all capitals.

## Background / Motivation

Readers often open the summary from a CI artifact, PR comment, or agent handoff
before they open the full report. The artifact should answer the first-read
questions quickly: what was evaluated, what is the verdict, which Areas and
Factors explain it, what findings matter most, and which recommendation ID is
the next handle for follow-up work.

## Scope

Covered: concise Markdown outline, key-detail labels, Area breakdown display,
selected-finding display, recommendation-action display, scope/limitation
display, and links to the full and machine-readable reports.

Deferred / non-goals: full requirement detail, full finding detail, full
recommendation audit trail, machine-readable state contracts, rating roll-up
semantics, and report gate behavior.

## Requirements

`report-summary.md` **MUST** be derived from the same assembled report model as
`report.md` and `report.json`. It **MUST NOT** reread evaluated source, recompute
ratings, invent findings, infer roll-ups, or choose new recommendations.

`report-summary.md` **MUST** link to `report.md` and `report.json`.

`report-summary.md` **MUST** use this top-level outline, in order:

1. key details under `# Quality Evaluation Summary`;
2. `## Verdict`;
3. `## Area Breakdown`;
4. `## Selected Findings`;
5. `## Recommended Actions`;
6. `## Scope & Limitations`.

The key details **MUST** include root Area, run when known, scope, rigor,
evaluation verdict, and links to the full and machine-readable reports.

The scope label for an unnarrowed evaluation **MUST** render as
`Full evaluation`.

The evaluation verdict **MUST** be the in-scope root Area's
Area-with-descendants rating. It **MUST NOT** be the Area-only rating unless the
recorded analysis makes those ratings equivalent.

The Area Breakdown section **MUST** show each in-scope Area's display path,
Area-only rating, Area-with-descendants rating, and Factor breakdown.

Human Markdown **SHOULD** label the Area Breakdown columns as `Path`, `Area`,
`+ Sub-Areas`, and `Factors`.

Area display paths **MUST** be absolute. The root Area renders as
`/ (<root title>)`; descendants render with a leading `/`, such as
`/Services/Payments/Webhooks`.

Areas with child Areas but no direct requirements **MUST** render the Area-only
rating as `(area group)` in human Markdown.

`not assessed` outcomes **MUST** render distinctly from rating levels.

Areas with no recorded Factor ratings **MUST** render an explicit empty-state in
the Factor breakdown.

The Selected Findings section **MUST** include only deterministic selected
findings from the assembled report model. Informational findings **MUST NOT** be
promoted into selected risks unless a documented selection rule includes them.

The Recommended Actions section **MUST** make active recommendation identifiers
prominent. When active recommendations exist, it **MUST** render a
`Recommendation ID` column with copyable stable identifiers and **MUST NOT**
present superseded recommendations as primary actions.

The Scope & Limitations section **MUST** state the effective scope and recorded
limitations. When no limitations are recorded, it may render a concise empty
state.

`report-summary.md` **MUST NOT** render detailed Requirements, full Findings, or
full Advice sections. Those belong in `report.md`.
