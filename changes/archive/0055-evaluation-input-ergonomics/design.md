# Self-describing evaluation record input - design doc

Design behind the
[Self-describing evaluation record input](../0055-evaluation-input-ergonomics.md)
change case and its [functional spec](spec.md). The spec says _what_ the record
surface must require; this doc says _how_ the Go code delivers it and why these
choices over the alternatives. Behavior and requirements live in the spec â€” this
doc does not restate them.

## Context

The write path is `internal/cli/evaluation.go` (the cobra wiring,
`newEvaluationRecordWriteCmd`) over `internal/evaluation/write.go`. Today each
write command calls `evaluation.WriteRecords(kind, runPath, raw)`, which
dispatches to `addAssessmentResults` / `setAnalyses` / `addRecommendations`. Each
of those does the same four things inline and in one pass:

1. `DecodeJSONList[T]` â€” strict decode (`DisallowUnknownFields`), object-or-array.
2. `validate<Kind>` â€” a sequence of `if â€¦ { return usagef(â€¦) }` checks that
   **returns on the first failure**.
3. build the `â€¦Record`, `marshalJSON` / `renderRecommendation`.
4. `writeNumbered` / `writeReplace` â€” mutate the run folder.

Two things make the surface opaque, both visible in this flow: there is no place
to stop after step 2 (so validation always implies a write), and the errors out
of steps 1â€“2 surface one at a time â€” step 1's come from `encoding/json` and name
Go types (`â€¦ .factorPaths of type evaluation.FactorPath`), step 2's already name
JSON keys at HEAD but short-circuit. The command help (`Short` only) says nothing
about the payload at all.

## Approach

### Split each write into decode â†’ validate â†’ plan â†’ commit

Refactor the three kind functions so the four inline steps become named stages,
with a hard seam between **plan** (pure: produces the intended `[](path, bytes)`
for the run) and **commit** (the only stage that touches disk). `WriteRecords`
gains a `WriteOptions{DryRun bool}` parameter; on `DryRun` it runs
decodeâ†’validateâ†’plan and returns, skipping commit. This single seam delivers
three requirements at once: the dry-run validates without persisting, the receipt
can report the paths it _would_ write, and the commit stage shrinks to "write
these bytes."

Path numbering moves into plan: `writeNumbered`'s directory scan splits into a
read-only `nextNumberedPath` (predict the path) used by plan and a thin write
used by commit. Under dry-run the predicted path is reported as-is.

### Aggregated, key-named validation

Replace first-error short-circuit with an accumulator. A small
`type ValidationError struct { Problems []FieldProblem }` (each problem: JSON
key, message, optional expected-type/enum) implements `error` and is wrapped in
the existing `UsageError`, so exit-code mapping (category 2) is unchanged. The
`validate<Kind>` functions append to the accumulator instead of returning, and
report all problems together; array payloads prefix each problem with `[i]`.

Step 1's decoder errors are folded into the same vocabulary before semantic
validation runs, in a shared `decodeRecord` helper:

- **Unknown fields, all at once.** Pre-decode into `map[string]json.RawMessage`
  and diff the keys against the struct's `json` tags, so every stray key is
  reported together with the allowed set â€” instead of `DisallowUnknownFields`
  surfacing them one per round-trip.
- **Type mismatches, in the caller's words.** Catch `*json.UnmarshalTypeError`,
  take its `.Field` (already the JSON key path) and translate `.Type` into a JSON
  shape description anchored to the canonical example â€” e.g.
  `factorPaths: expected an array of path arrays, e.g. [["reliability"]]` â€” never
  the Go type name.

### `-n/--dry-run`

`newEvaluationRecordWriteCmd` binds `-n/--dry-run` and passes
`WriteOptions{DryRun: true}` through. Valid payload â†’ exit 0; invalid â†’ the
`UsageError` path already maps to category 2. The `--json` receipt reuses
`WriteRecordReceipt` with a `"dryRun": true` marker and `paths` set to the
intended writes, so the dry-run is usable unattended. Human output states what
_would_ be written rather than "Wrote â€¦".

### Self-describing help

Each write command's `Long` carries a field table â€” name, type, required, enum â€”
and `Example` carries one complete valid payload. To keep the table from becoming
a second hand-maintained schema, it is generated by reflecting over the `*Input`
struct (json tag â†’ key and type) augmented with a small declarative
required/enum annotation map colocated with the struct; a test asserts the
generated table covers every struct field, so a new field can't ship
undocumented.

### One canonical example, golden-tested, shared everywhere

The drift-proofing requirement is met without a schema generator: keep **one
canonical example payload per kind** as a Go fixture, embed it in the command's
`Example` and in `resources/cli-quick-reference.md`, and assert in a test that
each fixture decodes-and-validates clean through the real
`WriteRecords(â€¦, DryRun)` path. The example is therefore validated by the same
code that rejects a bad payload â€” add a required field and the fixture fails the
test until the example (and the help/quick-ref that embed it) are updated. A
companion test extracts the fenced payloads from the shipped quick-ref and
asserts each **decodes-and-validates clean** through the real
`WriteRecords(â€¦, DryRun)` path, closing the skill-side drift the field run hit
without coupling the doc's formatting to a Go string literal.

### Skill reach and packaging

- `SKILL.md` drops the `../../specs/evaluation-records.md` citation and points
  record writing at `qualitymd evaluation <kind> add --help` and `--dry-run` as
  the authority; `modes/evaluate.md` adopts the discover-via-help /
  validate-via-dry-run flow.
- `scripts/check-npm-package.mjs` gains a guard that walks the shipped bundle's
  `.md` files, extracts relative links, and fails when one doesn't resolve inside
  the bundle.

### Coverage payload (SHOULD)

The `plan.md` coverage validation in `status` reuses the same accumulator/key
vocabulary, and `evaluation create` seeds a commented coverage stub in `plan.md`
so the shape is discoverable without a failed `status`.

## Alternatives

- **A standalone `evaluation <kind> schema` command.** Rejected per the spec's
  non-goal: help already carries the contract, and a new noun is one more surface
  to learn. If machine-readable JSON Schema is ever wanted, a `--schema` flag on
  the existing command is the smaller move.
- **A full reflective schema/example _generator_.** Rejected as over-built. A
  generator that synthesizes a valid example from types needs example _values_
  per field anyway, so it doesn't remove the hand-authored content â€” it just adds
  machinery. A single golden-tested fixture gets the same guarantee (the example
  is provably valid) for far less.
- **A descriptor-table-driven validator** (one table per kind driving messages,
  help, and example). Cleanest in theory and tempting because it unifies four
  things â€” but it's a rewrite of validation that the spec doesn't require, and it
  trades the readable per-field checks for a table interpreter. Deferred; the
  accumulator gets the same user-visible result over the existing checks.
- **Keep one-error-at-a-time, just fix the names.** Rejected: HEAD already fixed
  most of the names; the round-trips are the larger cost, and they only go away
  with aggregation.

## Trade-offs & risks

- **Dry-run numbering is a prediction.** `nextNumberedPath` reads the directory
  without locking, so a concurrent real write could take the predicted number.
  Acceptable for a validation affordance; the receipt describes intent, not a
  reservation.
- **Decoder type-error translation is best-effort.** `*json.UnmarshalTypeError`
  covers the common shape mistakes (the field run's `factorPaths`/`analyses`
  cases), but exotic decode failures may still read structurally. The canonical
  example in help is the backstop â€” a caller copies a known-good payload rather
  than decoding the error.
- **Reflection in the help table adds a little complexity** to an otherwise
  data-free package; the coverage test is what keeps it honest, and it stays
  contained to help rendering (the write path doesn't depend on it).

## Resolved decisions

- **Field-table depth: minimal.** The `Long` table carries name, type, required,
  and enum values only â€” no per-field gloss. The field run's pain was not knowing
  a field existed or was required, which those columns cover completely and the
  coverage test keeps complete; the canonical `Example` carries a real value for
  the non-obvious fields. Prose glosses are where help re-grows into a second,
  drift-prone copy of the spec, so constraint detail stays in
  [`specs/evaluation-records/`](../../../specs/evaluation-records/index.md).
- **Quick-ref drift test: validate-clean.** The test asserts each extracted
  payload decodes-and-validates through the real path, not byte- or
  semantic-equality. That catches the exact failure the field run hit â€” a payload
  the binary would reject shipping in the docs â€” while surviving harmless
  formatting differences that byte-equality would fail on. If a _value_ (not
  formatting) drift ever slips past it, the upgrade is normalized semantic-equality,
  not byte-equality.
