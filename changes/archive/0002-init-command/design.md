# init command — design doc

Design behind the [Specify the init command](../0002-init-command.md) change and
its [functional spec](spec.md). The spec fixes *what* `init` produces and how it
behaves; this doc covers *how* the code makes it so, on the established Go +
Cobra + Fang stack.

## Context

`init` must emit a commented target → factor → requirement skeleton with a seeded
rating scale and headed body stubs, write it to `QUALITY.md` (or a given path, or
stdout via `-`), and refuse to overwrite an existing file unless `--force` is
passed. The scaffold MUST itself be conformant — a freshly initialized file
passes `lint`. See the [spec](spec.md) for the full requirement set.

## Approach

### The scaffold is a static embedded asset

The skeleton is authored once as a real `QUALITY.md`-shaped file and embedded
into the binary with `//go:embed`, then written out verbatim. It is **not**
generated by marshalling a `spec.Spec` value to YAML.

The decisive reason is the *commented* part of "commented skeleton": the inline
YAML comments that guide the author, plus the Markdown body prose, do not
round-trip through `gopkg.in/yaml.v3` struct marshalling — comments would have to
be reconstructed via hand-built `yaml.Node` head/line-comment fields, and the
body has no struct at all. A static asset sidesteps that entirely: what we ship
is exactly what a human authored and can read, lint, and diff directly.

The asset lives at `internal/scaffold/skeleton.md` — deliberately **not** named
`QUALITY.md`, so it does not read as a quality model for the `internal/scaffold`
directory itself.

```go
// internal/scaffold/scaffold.go
package scaffold

import (
	_ "embed"
	"fmt"
	"os"
)

//go:embed skeleton.md
var skeleton []byte

// Bytes returns the raw scaffold for stdout emission.
func Bytes() []byte { return skeleton }

// Create writes the scaffold to path, refusing to overwrite unless force is set.
func Create(path string, force bool) error {
	flags := os.O_WRONLY | os.O_CREATE | os.O_EXCL
	if force {
		flags = os.O_WRONLY | os.O_CREATE | os.O_TRUNC
	}
	f, err := os.OpenFile(path, flags, 0o644)
	if err != nil {
		if os.IsExist(err) {
			return fmt.Errorf("%s already exists; pass --force to overwrite", path)
		}
		return err
	}
	defer f.Close()
	_, err = f.Write(skeleton)
	return err
}
```

This keeps `internal/cli` as thin wiring and the logic in an `internal/scaffold`
package, mirroring how `check` delegates to `spec`/`eval`/`report`.

### Overwrite protection is an atomic exclusive create

`Create` relies on `O_CREATE|O_EXCL` rather than a stat-then-write check: the
kernel fails the open with `EEXIST` if the file already exists, so there is no
time-of-check/time-of-use window where a file could appear between the check and
the write. `--force` swaps `O_EXCL` for `O_TRUNC` to overwrite in place. The
stdout path (`-`) never touches the filesystem and so bypasses this entirely.

### Command wiring

A new `internal/cli/init.go` adds `newInitCmd()`, registered in `root.go`
alongside `check`. It resolves the path argument, branches on `-`, and otherwise
delegates to `scaffold.Create`. The `--force` flag is a plain Cobra bool — no
divergence from stack defaults.

```go
func newInitCmd() *cobra.Command {
	var force bool
	cmd := &cobra.Command{
		Use:   "init [path]",
		Short: "Scaffold a starter QUALITY.md",
		Args:  cobra.MaximumNArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			path := "QUALITY.md"
			if len(args) == 1 {
				path = args[0]
			}
			if path == "-" {
				_, err := cmd.OutOrStdout().Write(scaffold.Bytes())
				return err
			}
			if err := scaffold.Create(path, force); err != nil {
				return err
			}
			fmt.Fprintf(cmd.ErrOrStderr(), "Created %s\nNext: qualitymd lint %s\n", path, path)
			return nil
		},
	}
	cmd.Flags().BoolVar(&force, "force", false, "overwrite an existing file")
	return cmd
}
```

The confirmation and the `Next:` footer go to **stderr**, leaving stdout clean so
`qualitymd init - > QUALITY.md` is never polluted — consistent with the CLI
spec's placement of next actions on stderr.

### A conformance test pins the asset

A test writes the embedded skeleton to a temp file and runs it through
`spec.Load`, asserting it parses and validates. This enforces the spec's
lint-clean requirement at the loader's current structural level and fails CI the
moment an edit to `skeleton.md` drifts out of conformance.

## Alternatives

- **Marshal a `spec.Spec` value to YAML.** Rejected: loses the guiding comments
  and the body prose (see [above](#the-scaffold-is-a-static-embedded-asset)). It
  would also couple the scaffold to the internal struct shape, so a struct change
  could silently alter author-facing output.
- **`text/template` to interpolate a title from the working directory.** Tempting
  — `title: <dirname>` reads better than a literal placeholder — but it adds a
  templating step for one field the author will overwrite anyway. Deferred; the
  static asset ships a literal placeholder `title` with a matching body heading.
- **Inline the skeleton as a Go string constant in `internal/cli`.** Rejected: a
  multi-hundred-byte Markdown document is far more maintainable as its own file
  an editor renders and dprint formats than as an escaped Go string literal.

## Trade-offs & risks

- **Two sources to keep in step.** The embedded `skeleton.md` and the format spec
  can drift — e.g. if the suggested rating scale changes. The conformance test
  catches structural drift; semantic drift (the skeleton stops being *good*
  example content) is a review concern, not a mechanical one.
- **`--force` overwrites non-atomically.** The default `O_EXCL` create path is
  atomic — it either creates the file or fails — but `--force` opens with
  `O_TRUNC`, so a write that fails midway leaves a truncated file, the original
  already discarded. A temp-file-and-rename would make overwrite atomic too; it is
  omitted as overkill for a scaffold the author is about to edit, but the
  asymmetry is worth recording.
- **No directory-target handling.** Passing a directory path resolves to whatever
  `os.OpenFile` does with it (an error). The spec defers directory semantics, so
  this is acceptable for now but may surprise; the error message should stay
  legible.

## Open questions

- **Shared next-actions machinery.** The footer is a hand-rolled stderr line for
  now. The [CLI spec](../../../specs/cli.md#conventions) defines a richer next-actions
  convention (stable `id`/`label`/`command`, suppressible via a quiet flag) that
  is not yet built. `init` should migrate to that shared mechanism when it lands
  rather than entrenching the ad-hoc footer.
