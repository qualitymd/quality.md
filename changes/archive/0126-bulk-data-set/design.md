# Bulk data set — design

Companion to the [Bulk data set](../0126-bulk-data-set.md) change case and its
[functional spec](spec.md). The spec defines the array-only batch contract; this
design covers how the existing `SetData` path changes to validate every payload,
stage writes, and return one receipt.

## Context

Today `evaluation.SetData` does one linear operation: decode one JSON object,
load `model-snapshot.md`, validate that object, derive one `data/**` path, and
write that file. The CLI wrapper reads stdin as bytes, calls `SetData`, and
prints one path.

0126 keeps the public command surface but changes stdin from one object to a
non-empty array. That turns the implementation from one payload pipeline into a
batch pipeline with three important properties:

- model snapshot loading happens once;
- every payload is validated and routed before any write;
- successful output describes the batch, not one artifact.

## Approach

Introduce an internal candidate type for a fully validated pending write:

```go
type dataWriteCandidate struct {
    Index     int
    Kind      DataKind
    Path      string
    Canonical []byte
}
```

`SetData` becomes a batch entry point:

1. Decode stdin as `[]map[string]any` with `json.Decoder.UseNumber`.
2. Reject non-array input, trailing JSON, and `[]` as usage errors.
3. Load the run's `model-snapshot.md` once.
4. For each element, resolve `kind`, reject `EvaluationOutputResult`, validate
   against the loaded model, canonicalize JSON, and derive the target path.
5. Collect every per-element validation failure with the array index and field
   context where available.
6. Reject duplicate target paths inside the candidate list before any write.
7. If `--dry-run` is set, return the batch receipt without writing.
8. Stage and commit every candidate write.

Validation helpers stay mostly unchanged. The existing single-payload path is
split into small helpers (`candidateForPayload`, `validateCandidateBatch`,
`writeCandidateBatch`) so tests can assert the batch behavior without duplicating
payload construction.

## Diagnostics

Use a usage-error aggregate for batch validation failures. Each entry is a plain
message prefixed with `payload[<index>]`, for example:

```text
payload[2].subject.areaId does not resolve in the model
payload[5]: EvaluationOutputResult is generated by evaluation report build
```

The aggregate still maps to CLI usage exit `2` through the existing
`evaluation.UsageError` path. A malformed envelope, empty array, duplicate target
path, and payload validation failures are all usage errors. A missing or
unparseable `model-snapshot.md` remains an internal evaluation error for the
whole batch.

## Write Staging

The write phase starts only after every candidate is valid and path-unique.

For normal writes, stage all bytes first under a temporary directory inside the
run folder, preserving the target relative path under that temp root. If staging
fails, remove the temp directory and leave `data/` untouched.

Commit staged files with tracked rollback:

1. For each target, create its parent directory.
2. If the target exists, move it to a backup path under the temp directory.
3. Move the staged file into place.
4. Track whether the target was newly created or replaced.

If a commit step returns an error, best-effort rollback removes newly created
targets and restores backups before returning the error. This gives normal
command failures no partial batch effect. It is not a crash-consistency protocol:
process termination or machine failure during the final commit window can still
leave filesystem state between steps. That risk is acceptable for this CLI phase;
the spec's critical guarantee is that validation failures and ordinary write
errors are not allowed to silently produce a half-valid batch.

## Receipt

Replace the single-path receipt with a batch receipt:

```json
{
  "schemaVersion": 2,
  "count": 2,
  "writes": [
    {
      "index": 0,
      "kind": "RequirementAssessmentResult",
      "path": "data/areas/root/requirements/has-tests/requirement-assessment-result.json"
    },
    {
      "index": 1,
      "kind": "RequirementRatingResult",
      "path": "data/areas/root/requirements/has-tests/requirement-rating-result.json"
    }
  ],
  "dryRun": true,
  "nextActions": []
}
```

`writes` is in input order and is the only per-payload receipt list. `count`
equals `len(writes)`. The CLI human form prints `Wrote <n> payloads` or
`Would write <n> payloads`, followed by the paths in input order. The receipt
keeps the existing next action pointing to evaluation status.

## Spec Response

- Input format requirements are handled by the new array decoder and non-empty
  check.
- Validation and atomicity are handled by the candidate pass, aggregate usage
  error, single model-snapshot load, duplicate-path preflight, and staged write
  commit with rollback.
- Idempotent overwrites are preserved because existing targets are replaced by
  the candidate canonical JSON.
- Dry-run returns the same receipt shape and candidate path set without entering
  the staging/commit phase.
- The CLI-owned `EvaluationOutputResult` stays rejected by the same kind gate,
  applied to every candidate.

## Alternatives

- **Accept both object and array input.** Rejected because the change deliberately
  removes the old transport contract instead of maintaining two paths.
- **NDJSON.** Rejected because the agent assembles the batch as one structure,
  the CLI needs whole-batch validation, and JSON arrays compose directly with
  `model list --json` output.
- **Validate and write one element at a time.** Rejected because any later failure
  could leave a partial batch and would not satisfy the all-or-nothing contract.
- **Copy and swap the entire `data/` tree.** This improves crash recovery but is
  heavier, risks losing concurrent unrelated writes, and is not needed for the
  current single-agent CLI workflow.

## Trade-offs & Risks

- Aggregated diagnostics are text, not a structured error JSON contract. That
  matches the current CLI error posture; a structured error envelope can be added
  later if the CLI gets one generally.
- Rollback is best-effort for ordinary write errors, not crash-safe. The CLI does
  not currently promise crash-safe multi-file transactions elsewhere.
- Receipt shape is breaking for `--json` consumers of `data set`. This is
  intentional early-alpha cleanup and is documented in the durable spec update.
